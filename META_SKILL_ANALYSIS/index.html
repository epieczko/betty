    meta.skill Agent - Detailed Analysis & Shortcomings¶

    Date: 2025-11-02 Agent: agents/meta.skill/agent.yaml Purpose: Creates complete, functional skills from natural language descriptions Status: draft (untested end-to-end)

    What meta.skill SHOULD Do¶

    From the agent definition:

    description: |
  Creates complete, functional skills from natural language descriptions.

  This meta-agent transforms skill descriptions into production-ready skills with:
  - Complete skill.yaml definition
  - Python implementation stub with proper structure
  - Test template with example tests
  - README documentation
  - Registry registration

  Ensures skills follow Betty Framework conventions and are ready for use in agents.

    Available Skills: 

    skills_available:
  - skill.create      # Scaffolds directory and files
  - skill.define      # Validates skill manifest
  - artifact.define   # Generate artifact metadata (KEY FOR VALIDATION!)

    Identified Shortcomings¶

    1. No Explicit Artifact Validation Step ‚ö†Ô∏è¶

    Current System Prompt (Step 2): 

    2. **Generate skill.yaml** - Create complete definition with ALL required fields
   - Artifact metadata (produces/consumes)

    Problem: - Says to include artifact metadata but doesn't say HOW to validate it - Doesn't specify to call artifact.define skill - No guidance on checking artifact types exist in registry - No error handling if artifact type is invalid

    Should Be: 

    2. **Validate Artifact Types** - Before generating skill.yaml
   - Extract all artifact types from skill description
   - For EACH type in produces/consumes:
     a. Call artifact.define to verify type exists in registry
     b. Retrieve file_pattern, content_type, schema from registry
     c. Verify schema file exists (if specified)
   - If type doesn't exist:
     ‚Üí Search for similar types (handle singular/plural, typos)
     ‚Üí ERROR: "Artifact type 'data-flow-diagram' not found. Did you mean 'data-flow-diagrams'?"
     ‚Üí HALT skill creation until resolved

    Evidence: 

    # System prompt mentions artifact.define but never says when to call it:
skills_available:
  - artifact.define    # Generate artifact metadata

    But the workflow never explicitly calls it!

    2. Relies on Broken registry.update ‚ùå¶

    Current System Prompt (Step 6): 

    6. **Register Skill** - Update registry
   - Add to registry/skills.json
   - Include all metadata
   - Validate registration

    Problem: - registry.update uses SkillManifest Pydantic model - SkillManifest expects inputs: List[str] - Actual skills use inputs: List[Object] - Registration will ALWAYS fail with validation error!

    What Happens: 

    meta.skill ‚Üí skill.create ‚Üí skill.define ‚Üí registry.update
                                                ‚Üì
                                         ‚ùå Pydantic ValidationError:
                                            "Input should be a valid string"

    This means meta.skill CANNOT complete step 6 successfully!

    3. Vague Error Handling ü§∑¶

    Current System Prompt: 

      ## Quality Standards

  - ‚úÖ Follows Betty conventions
  - ‚úÖ All required fields in skill.yaml: name, version, description, inputs, outputs, status
  - ‚úÖ Proper artifact metadata
  - ‚úÖ Clean, documented code
  - ‚úÖ Test template included
  - ‚úÖ SKILL.md with markdown header and examples
  - ‚úÖ Registered in registry
  - ‚úÖ Passes validation tests

    Problem: - Says "passes validation tests" but what if it doesn't? - No guidance on what to do when: - Artifact type not found - Registry update fails - Duplicate skill name detected - Schema file missing - Permission conflicts

    Should Have: 

      ## Error Handling & Recovery

  - **Artifact Type Not Found:**
    ‚Üí Call artifact.define to search for similar types
    ‚Üí Suggest: "Artifact 'data-model' not found. Did you mean: 'logical-data-model', 'physical-data-model', 'enterprise-data-model'?"
    ‚Üí Ask user to confirm or provide correct type
    ‚Üí Do NOT proceed with invalid artifact types

  - **Registry Update Fails:**
    ‚Üí Report specific Pydantic error
    ‚Üí Provide manual registration workaround
    ‚Üí Log issue for framework team

  - **Duplicate Skill Name:**
    ‚Üí Check existing version in registry
    ‚Üí Offer to: (a) version bump, (b) rename, (c) cancel
    ‚Üí Ensure user intent before overwriting

  - **Schema File Missing:**
    ‚Üí Warn: "Schema file schemas/artifacts/threat-model-schema.json not found"
    ‚Üí Ask if schema should be: (a) created, (b) removed from reference, (c) ignored
    ‚Üí Continue with warning, don't block

    5. Missing Validation Hooks ü™ù¶

    Current Workflow: 

    skill.create ‚Üí skill.define ‚Üí registry.update

    Problem: - No validation between steps - Can't catch errors early - All-or-nothing approach

    Better Workflow: 

    1. Validate artifact types (NEW STEP)
   ‚Üì (artifact.validate.types)
2. Generate skill.yaml with validated metadata
   ‚Üì (skill.create)
3. Validate manifest schema
   ‚Üì (skill.define)
4. Pre-validate registry entry (dry-run)
   ‚Üì (registry.validate - NEW SKILL)
5. Register skill
   ‚Üì (registry.update)
6. Post-validate discoverability
   ‚Üì (agent.compose dry-run to confirm skill findable)

    Each step has validation gates!

    6. No Artifact Flow Analysis üìä¶

    meta.skill should answer: - What artifacts does this skill produce? - What skills consume those artifacts? - What artifacts does this skill need? - What skills produce those artifacts? - Is there a complete artifact flow?

    Example: 

    Creating skill: threat.model.generate

Artifact Flow Analysis:
‚úÖ Produces: threat-model
   ‚Üí Consumed by: (none yet - this is the first!)
   ‚Üí Recommendation: Consider creating compliance.matrix.generate
                     to consume threat-model

‚úÖ Consumes: architecture-overview
   ‚Üí Produced by: (need to create architecture.designer agent)
   ‚Üí Warning: No producer exists yet

‚ö†Ô∏è  Consumes: data-flow-diagrams
   ‚Üí Produced by: (none found)
   ‚Üí Action Required: Create data.flow.diagram.generate skill
                      OR remove from consumes if not critical

    This would prevent orphaned skills!

    7. Doesn't Verify Skill Discoverability üîç¶

    After registration, meta.skill should verify:

    # Step 7: Verify Discoverability
def verify_skill_registered():
    """Confirm skill can be discovered by agents."""

    # Check 1: Skill in registry
    registry = load_registry()
    skill = find_skill_by_name(registry, "threat.model.generate")
    assert skill is not None, "Skill not found in registry!"

    # Check 2: Has artifact_metadata
    assert "artifact_metadata" in skill, "Missing artifact_metadata!"

    # Check 3: Discoverable by artifact type
    for artifact in skill["artifact_metadata"]["produces"]:
        discovered = find_skills_by_artifact_type(artifact["type"])
        assert "threat.model.generate" in discovered, \
            f"Skill not discoverable by artifact type {artifact['type']}!"

    # Check 4: agent.compose can find it
    result = agent.compose(required_artifacts=["threat-model"])
    assert "threat.model.generate" in result["recommended_skills"], \
        "agent.compose cannot discover skill!"

    print("‚úÖ Skill successfully registered and discoverable!")

    Currently: meta.skill just hopes registration worked

    8. No Dependency Resolution üîó¶

    skill.yaml allows dependencies: 

    dependencies:
  - PyYAML
  - jsonschema
  - custom-library

    meta.skill should: - Check if dependencies are available - Suggest installation commands - Warn about version conflicts - Validate Python package names

    Currently: Accepts any string, no validation

    Summary of Shortcomings¶

    Critical (Blocks Functionality) üî¥¶

    1. No artifact type validation - Can create skills with invalid artifact types
    2. Broken registry.update dependency - Cannot register skills with detailed inputs
    3. Untested end-to-end - Don't know if it works at all

    High Priority (Quality Issues) üü°¶

    1. Vague error handling - Fails without helpful guidance
    2. No validation hooks - Can't catch errors early
    3. No discoverability verification - Doesn't confirm skill is usable

    Medium Priority (Enhancements) üü¢¶

    1. No artifact flow analysis - Misses orphaned skills
    2. No dependency resolution - Accepts invalid dependencies
    3. Poor template quality - Generated code has too many TODOs
    4. No feedback loop - Can't interact with user to fix issues

    Recommended Improvements¶

    Phase 1: Make It Work (Week 1)¶

    1. Add explicit artifact validation step
    2. Update system prompt with Step 1.5: "Validate Artifact Types"
    3. Specify calling artifact.define for each type

    4. Add error handling for missing types

    5. Work around registry.update bug

    6. Until SkillManifest is fixed, document manual registration
    7. OR create registry.add.skill that bypasses validation

    8. OR fix SkillManifest to accept Union[List[str], List[Object]]

    9. Test end-to-end

    10. Fix PYTHONPATH issues
    11. Run meta.skill with test skill description
    12. Document what works and what doesn't

    Phase 2: Make It Robust (Week 2)¶

    1. Add comprehensive error handling
    2. Specific recovery for each error type
    3. Clear user guidance

    4. Graceful degradation

    5. Add validation gates

    6. Pre-validate artifacts
    7. Pre-validate registry entry (dry-run)

    8. Post-validate discoverability

    9. Verify discoverability

    10. Test agent.compose can find skill
    11. Confirm artifact_metadata in registry
    12. Check artifact flow completeness

    Phase 3: Make It Smart (Week 3)¶

    1. Add artifact flow analysis
    2. Show what consumes skill's outputs
    3. Show what produces skill's inputs

    4. Warn about gaps

    5. Validate dependencies

    6. Check Python package names
    7. Suggest installation commands

    8. Warn about conflicts

    9. Improve templates

    10. Parse skill.yaml to generate proper CLI

      • Use iterative reasoning mode

      • Iterate until skill is correct

    Open Questions¶

    3. Workaround: Faster, but technical debt

    5. None: Fully autonomous (current)
    6. Some: Confirm fixes for errors (proposed)

    8. Should meta.skill create artifact types if missing?

    9. Pro: Enables new domains quickly
    10. Con: Could lead to artifact sprawl
